# Unit Reference Docs & measurementSystem Field

## What & Why
Two things together: (1) add a `measurementSystem` field to every unit in all JSON data files so future tooling always knows which system a unit belongs to; (2) generate three hand-QA-able markdown reference documents in `docs/measures/` from that data. All three documents regenerate automatically on every merge.

## Done looks like
- `UnitDefinition` in `client/src/lib/units/unitDefinition.ts` has an optional `measurementSystem` field typed against a new `MeasurementSystem` string-union or const.
- Every unit in every `client/src/data/conversion/*.json` file has a `measurementSystem` value. Units that cannot be classified use `"NONE"`.
- `docs/measures/units.md` — every unit, one row per unit: symbol | en name | quality/qualities | base SI symbol.
- `docs/measures/compare-all.md` — one section per quality with a table: symbol | en name | base SI value ("1 [sym] = [factor] [SI sym]").
- `docs/measures/units-by-system.md` — one section per measurement system: symbol | en name | base SI value. Sections use human-readable system names.
- All three files are produced by `scripts/generate-unit-docs.mjs` (plain Node.js, no extra deps) and regenerated by `scripts/post-merge.sh`.
- Initial copies of all three files are committed as part of this task.

## Measurement system taxonomy to implement
Define these values as a TypeScript string-union or const object. The executor should use this list as the starting point and may add entries if the data warrants it.

**Modern standards**
- `SI` — SI metric (base + derived SI units)
- `US_CUSTOMARY` — US everyday customary (replaces current US_COMMON)
- `US_CUSTOMARY_FLUID` — US fluid measures (replaces US_COMMON_FLUID)
- `US_CUSTOMARY_DRY` — US dry measures (replaces US_COMMON_DRY)
- `US_SURVEY` — US survey system (survey foot, Gunter's chain, link, rod, furlong, league)
- `US_FDA` — US FDA nutrition-labeling units (if present in the data)
- `IMPERIAL` — British Imperial
- `IMPERIAL_SURVEY` — British survey (Gunter's chain system as used in Britain)
- `APOTHECARIES` — Apothecaries' system (scruple, dram apothecaries')
- `TROY` — Troy weight (grain, pennyweight, troy oz, troy lb)
- `ASTRONOMICAL` — Astronomical units (AU, light-year, parsec, etc.)

**French pre-metric system (pied-du-roi)**
This single root system covers both the Didot typographic tradition and Swiss watchmaking, since all derive from the French *pied-du-roi* hierarchy (pied → 12 pouces → 12 lignes each → 12 douzièmes each):
- `FRENCH_TRADITIONAL` — French pre-metric (*pied-du-roi*) system. Includes:
  - *Typographic branch*: Didot point (= 1/72 French pouce ≈ 0.376 mm), Cicero (= 12 Didot points ≈ 4.512 mm), Ligne (= 1/12 French pouce ≈ 2.256 mm)
  - *Watchmaking branch*: Ligne (‴) — same unit, used for watch movement diameter; douzième (= 1/12 ligne) if present in data
  - *General*: French livre, French pied, French pouce if present in data

Note: **Cicero is currently labelled `IMPERIAL` in `typography.json` — this is incorrect and must be corrected to `FRENCH_TRADITIONAL`.**
Note: The localization files already contain a "Didot" key, suggesting the Didot point unit was planned. The executor should add it to `typography.json` if absent (factor ≈ 0.000376065 m). The Ligne (‴) should also be added if absent (factor ≈ 0.0022558 m).

**Anglo-American typographic system**
- `TYPOGRAPHIC_AMERICAN` — American/British traditional typographic system established 1886/1903: traditional pica point (≈ 0.3515 mm = 1/72.27"), traditional pica (≈ 4.217 mm). Digital descendants (CSS point = 1/72", px, em, twip) are `DOMAIN`, not this system.

**Regional / traditional**
- `JAPANESE` — Japanese traditional (shakkan-hō: shaku, sun, ken, jō, ri, momme, go, sho, koku, etc.)
- `KOREAN` — Korean traditional (ja, ri-kr, don, geun, hop, doe, etc.)
- `CHINESE` — Chinese traditional (cun, chi, zhang, li, mace, tael, jin, dan, etc.)
- `HONG_KONG` — Hong Kong traditional (chek, catty, picul, etc.)
- `SOUTH_ASIAN` — South Asian traditional (hath, gaj, ratti, tola, seer, maund, etc.)
- `THAI` — Thai traditional (baht, etc.)
- `GERMAN` — German historical (pfund, etc.)
- `RUSSIAN` — Russian traditional (arshin, etc.)
- `SPANISH` — Spanish traditional (vara, etc.)

**Ancient**
- `ANCIENT_EGYPTIAN` — Ancient Egyptian (digit, royal cubit, palm, span, etc.)
- `ANCIENT_GREEK` — Ancient Greek (stade, etc.)
- `ANCIENT_ROMAN` — Ancient Roman (pace, etc.)
- `ANCIENT_HEBREW` — Biblical/Hebrew (if present in data)

**Domain-specific (no governing measurement system)**
- `DOMAIN` — Industry/domain-specific with no measurement system body: TEU/DEU shipping containers, CSS pixel, CSS point, em, twip, typography points as digital conventions

**Fallback**
- `NONE` — Present but cannot be assigned to a known system

## Classification notes for the executor
- Chain (Gunter), Link (Gunter), Rod/Pole/Perch, Furlong → `US_SURVEY` (currently mislabelled `US_COMMON`)
- Fathom → `IMPERIAL` (nautical, codified in Imperial)
- Hand, Ell (English) → `IMPERIAL`
- League → `IMPERIAL` or `US_CUSTOMARY` depending on which variant is in the data; research each
- Pennyweight, Grain (troy), Troy oz/lb → `TROY`
- Scruple, Dram (Apothecaries') → `APOTHECARIES`
- Cicero → `FRENCH_TRADITIONAL` (fix the current `IMPERIAL` label)
- Didot point, Ligne (typographic/watchmaking) → `FRENCH_TRADITIONAL`
- Traditional pica, traditional pica point → `TYPOGRAPHIC_AMERICAN`
- CSS pt, px, em, twip → `DOMAIN`
- TEU, DEU → `DOMAIN`
- Pfund (German) → `GERMAN`; Livre (French) → `FRENCH_TRADITIONAL`
- Arshin (Russia) → `RUSSIAN`; Vara (Spain) → `SPANISH`
- All Japanese, Korean, Chinese, HK, South Asian, Thai archaic units → their respective regional system (not `SI_BASE` as currently coded)
- Baht (Thailand) → `THAI`
- Carat (Metric) → `SI` (it is defined relative to the gram by international agreement)

## Tasks
1. **Define `MeasurementSystem` type** — Add the taxonomy above as a TypeScript string-union or const in `client/src/lib/units/unitType.ts` (or a new `measurementSystem.ts` file). Add the optional `measurementSystem` field to `UnitDefinition` in `client/src/lib/units/unitDefinition.ts`.
2. **Annotate all JSON files** — Go through every file in `client/src/data/conversion/` and add `"measurementSystem"` to every unit entry. Fix the `Cicero` labelling. Add the Didot point unit and Ligne unit to `typography.json` if they are absent (with appropriate factor values from the research notes above). Archaic/regional units that currently have `unitType: "SI_BASE"` must be assigned their real system.
3. **Write the generator script** — Create `scripts/generate-unit-docs.mjs` reading all JSON files and `client/src/data/localization/units/en.json`. Produce all three documents. Units with `offset` or `mathFunction` fields get a footnote flagging non-linear conversion. The by-system document groups by `measurementSystem`, using human-readable section headings.
4. **Hook into post-merge** — Append `node scripts/generate-unit-docs.mjs` to `scripts/post-merge.sh` after the existing task-plan copy step. Create `docs/measures/README.md` explaining the files and how they are generated.
5. **Generate initial copies** — Run the script and commit the three initial documents.

## Out of scope
- Changing how the UI uses `unitType` (that field stays as-is; `measurementSystem` is purely additive).
- Validating or correcting conversion factor values (except for newly added Didot/Ligne units).
- Locale-specific variants of the docs beyond English (en).
