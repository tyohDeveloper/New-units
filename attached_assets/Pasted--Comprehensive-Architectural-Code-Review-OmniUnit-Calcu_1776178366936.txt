# Comprehensive Architectural & Code Review: OmniUnit & Calculator

**Repository:** OmniUnitConverter-Calculator  
**Version:** 3.2.1.1  
**Tech Stack:** React 19 + TypeScript 5.6 + Vite 7 + Tailwind CSS v4 + Radix UI  
**Date:** 2026-04-14  
**Reviewer:** Claude Opus 4.6

---

## Executive Summary

OmniUnit & Calculator is a sophisticated, client-only single-page application that serves as a universal unit converter with an integrated dimensional-analysis calculator. The project demonstrates strong domain modeling (dimensional formulas, SI derived units, cross-domain matching) and impressive breadth of unit coverage (90+ categories, 12 languages, 8+ number formats). The test suite is robust (1,536 passing tests, zero failures, clean TypeScript compilation).

The architecture has a clear tension: the project started with a clean reducer/context/selector pattern but the primary orchestrator component (`UnitConverterApp.tsx`, 1,676 lines) accumulated significant business logic and state that belongs in the reducer layer. This "god component" is the project's biggest structural issue. Several other concerns -- duplicate data structures, accessibility gaps, unused dependencies, and extreme prop drilling -- compound the problem but are individually tractable.

**Overall Assessment: B+** -- Solid domain logic, excellent test coverage, good modular utilities. Held back by the orchestrator's complexity and some web-standards gaps.

## PART 1: ARCHITECTURAL REVIEW

### 1. Project Structure

**Severity: Minor**

The directory layout follows a reasonable feature-based organization:

```
client/src/
  components/         # Shared UI + unit-converter sub-components
    ui/               # 60+ shadcn/ui primitives
    unit-converter/   # State management (reducers, actions, selectors, hooks)
  features/           # Feature-level components
    unit-converter/
      app/            # UnitConverterApp orchestrator
      components/     # ConverterPane, CalculatorPane, DirectPane
  data/               # Static JSON (conversion data, localization)
  lib/                # Core business logic
    calculator/       # 45+ dimensional analysis utilities
    units/            # 30+ unit system utilities
  pages/              # Route-level pages
```

**Findings:**

- **Good:** Clear separation between data (`data/`), business logic (`lib/`), state management (`components/unit-converter/state/`), and UI (`features/`, `pages/`).
- **Good:** The fine-grained decomposition of calculator utilities (`lib/calculator/`) into ~45 single-purpose files is excellent for testability and tree-shaking.
- **Issue:** The `components/unit-converter/` directory conflates UI components, state management (reducers, actions, selectors), and hooks. A `state/` directory exists within it, but the boundary between "component" and "state management" is blurred. Consider elevating `state/` to a top-level concern.
- **Issue:** The `components/ui/` directory contains 60+ shadcn/ui primitives, many of which appear unused (e.g., `calendar.tsx`, `carousel.tsx`, `avatar.tsx`, `chart.tsx`, `drawer.tsx`, `menubar.tsx`, `navigation-menu.tsx`, `progress.tsx`, `resizable.tsx`, `sidebar.tsx`). This is scaffolding bloat from the initial setup.

### 2. Architecture Pattern

**Severity: Major**

The app uses a **Context + useReducer** pattern with domain-segregated reducers -- a well-chosen architecture for a client-only SPA of this complexity. The pattern is applied in `ConverterContext.tsx`:

```typescript
// client/src/components/unit-converter/context/ConverterContext.tsx:45-64
function appReducer(state: AppState, action: AppAction): AppState {
  switch (action.domain) {
    case 'converter': { ... }
    case 'calculator': { ... }
    case 'rpn': { ... }
    case 'uiPrefs': { ... }
  }
}
```

**However, this pattern is not consistently applied.** The `UnitConverterApp.tsx` component (1,676 lines) manages its own parallel state via `useState` that duplicates or extends what the reducers should own:

```typescript
// client/src/features/unit-converter/app/UnitConverterApp.tsx:73-139
// These useState calls duplicate reducer state
const [activeCategory, setActiveCategory] = useState<UnitCategory>('length');
const [fromUnit, setFromUnit] = useState<string>('');
const [toUnit, setToUnit] = useState<string>('');
const [fromPrefix, setFromPrefix] = useState<string>('none');
const [toPrefix, setToPrefix] = useState<string>('none');
const [inputValue, setInputValue] = useState<string>('1');
const [result, setResult] = useState<number | null>(null);
const [precision, setPrecision] = useState<number>(4);
const [calculatorPrecision, setCalculatorPrecision] = useState<number>(4);
const [comparisonMode, setComparisonMode] = useState<boolean>(false);
const [calculatorMode, setCalculatorMode] = useState<'simple' | 'rpn'>('rpn');
const [shiftActive, setShiftActive] = useState(false);
// ... 15+ more useState calls
```

The `ConverterContext` with its 4 reducers, 42 actions, and 31 selectors was clearly designed to own this state -- but `UnitConverterApp` bypasses it entirely, using raw `useState` instead. This creates two parallel state systems and defeats the purpose of the reducer architecture.

**Recommendation:** Migrate all state from `UnitConverterApp`'s `useState` calls into the existing reducers. The reducers, actions, and selectors are already defined and ready; they just aren't being used by the main component.

### 3. Separation of Concerns

**Severity: Major**

- **Business logic in the orchestrator:** `UnitConverterApp.tsx` contains conversion logic, formatting logic, clipboard handling, dimension mapping, paste parsing, and calculator computation -- all of which should live in the `lib/` layer or reducer middleware. For example, the `getCategoryDimensions` function (lines 325-368) is a 40-line inline function that duplicates `CATEGORY_DIMENSIONS` from `lib/units/categoryDimensions.ts`.

- **Duplicate dimension maps:** The category-to-dimension mapping exists in at least three places:
  1. `UnitConverterApp.tsx:326` (`getCategoryDimensions` -- inline, 73 categories)
  2. `lib/units/categoryDimensions.ts:9` (`CATEGORY_DIMENSIONS` -- canonical source)
  3. `lib/conversion-data.ts:507` (`getCategoryDimensionsForParse` -- another inline copy)
  
  These can drift. The canonical source should be used everywhere.

- **Prop drilling is extreme.** `ConverterPane` receives 35 props, `CalculatorPane` receives 68 props. This is a direct consequence of keeping state in the parent rather than in context:

```typescript
// client/src/features/unit-converter/app/UnitConverterApp.tsx:1533-1580
// ConverterPane receives 35 props passed one by one
<ConverterPane
  activeTab={activeTab}
  activeCategory={activeCategory}
  fromUnit={fromUnit}
  toUnit={toUnit}
  fromPrefix={fromPrefix}
  toPrefix={toPrefix}
  inputValue={inputValue}
  result={result}
  precision={precision}
  comparisonMode={comparisonMode}
  // ... 25 more props
/>
```

If the context/reducer pattern were used, child components would `useConverterContext()` directly instead of receiving 35+ props.

### 5. Scalability

**Severity: Suggestion**

Adding new unit categories is already straightforward:

1. Create a new JSON file in `data/conversion/` (e.g., `luminance.json`)
2. Import it in `conversion-data.ts` and add to `CONVERSION_DATA`
3. Add the category to the `UnitCategory` union type
4. Add the category to `CATEGORY_GROUPS` in `UnitConverterApp.tsx`
5. Add dimension mapping in `categoryDimensions.ts` (and the two duplicate maps)

Steps 4-5 are the friction points. The `CATEGORY_GROUPS` array (lines 167-176) and the three copies of the dimension map mean every new category requires changes in 4+ locations.

**Recommendation:** Generate `CATEGORY_GROUPS` and `getCategoryDimensions` from `CATEGORY_DIMENSIONS` to make the canonical source the single point of truth.

## PART 2: CODE REVIEW

### 1. Code Quality

**Severity: Major (UnitConverterApp), Minor (elsewhere)**

**The good:**
- Utility functions in `lib/calculator/` and `lib/units/` are exemplary: single-purpose, well-typed, named clearly
- Reducers are clean and follow standard patterns
- Type definitions (`DimensionalFormula`, `CalcValue`, `UnitDefinition`) are precise and expressive
- The `parseUnitText` function handles impressive edge cases (scientific notation, Unicode superscripts, SI symbols)

**The bad:**

`UnitConverterApp.tsx` (1,676 lines) is a god component containing:
- 30+ `useState` declarations
- 15+ `useEffect` hooks
- 20+ handler functions (copy, paste, format, parse, swap, clear, etc.)
- The full JSX render tree

Many functions within it are not memoized and are recreated on every render. Only 3 uses of `useCallback` exist in the entire 1,676-line file (lines 495, 506, and one other). Zero uses of `useMemo`.

**Naming inconsistency:**
```typescript
// Mixed naming for the same concept:
const normalizeMassUnit = normalizeMassUnitHelper;   // line 141 - alias
const normalizeMassValue = normalizeMassValueLib;     // line 142 - alias
const applyPrefixToKgUnit = applyPrefixToKgUnitLib;  // line 143 - alias
```
These aliases add indirection without value.

**DRY violations:**
The dimension-mapping code from `parsed.dimensions` to `newExponents` is repeated verbatim 3 times:
- `UnitConverterApp.tsx:229-238` (paste handler)
- `UnitConverterApp.tsx:541-552` (custom paste handler)
- `DirectPane.tsx:82-89` (blur handler)

```typescript
// This exact block appears 3 times:
if (parsed.dimensions.length) newExponents.m = parsed.dimensions.length;
if (parsed.dimensions.mass) newExponents.kg = parsed.dimensions.mass;
if (parsed.dimensions.time) newExponents.s = parsed.dimensions.time;
if (parsed.dimensions.current) newExponents.A = parsed.dimensions.current;
if (parsed.dimensions.temperature) newExponents.K = parsed.dimensions.temperature;
if (parsed.dimensions.amount) newExponents.mol = parsed.dimensions.amount;
if (parsed.dimensions.intensity) newExponents.cd = parsed.dimensions.intensity;
if (parsed.dimensions.angle) newExponents.rad = parsed.dimensions.angle;
if (parsed.dimensions.solid_angle) newExponents.sr = parsed.dimensions.solid_angle;
```

This should be a utility function: `dimensionsToExponents(dims: DimensionalFormula): Record<string, number>`.

### 3. Type Safety

**Severity: Minor**

TypeScript is used effectively overall. The project compiles cleanly with `strict: true` and zero errors.

**Strengths:**
- `UnitCategory` is a discriminated union of 73+ string literals -- prevents typos
- `AppAction` uses tagged unions with `domain` discriminant -- type-safe dispatch
- `DimensionalFormula` is properly typed with optional number fields
- The `CalcValue` interface captures both the numeric value and its dimensional context

**Weaknesses:**

- **`as unknown as` type escape** in `conversion-data.ts:281`:
  ```typescript
  function asCategoryDefinition(raw: RawCategoryJson): CategoryDefinition {
    return raw as unknown as CategoryDefinition;
  }
  ```
  This bypasses type checking entirely. A runtime validation (or at least a type guard) would be safer since the JSON data is user-authored.

- **Stringly-typed identifiers:** Unit IDs and prefix IDs are plain `string` throughout:
  ```typescript
  fromUnit: string;   // Could be a branded type
  toUnit: string;     // Could be a branded type
  fromPrefix: string; // Should be PrefixId
  ```
  Consider branded types or at least a `PrefixId` type alias to prevent mixing up unit IDs with prefix IDs with arbitrary strings.

- **Missing exhaustive check in appReducer** (line 45-64): The `switch` on `action.domain` has no `default` case. TypeScript's control flow analysis handles this, but an explicit `default: { const _: never = action; return state; }` pattern would catch future bugs if a new domain were added.

### 4. Performance

**Severity: Major**

- **No memoization in the orchestrator.** `UnitConverterApp.tsx` has 1,676 lines of component body that re-execute on every state change. With 30+ state variables, any change triggers a full re-render. Key concerns:

  - `getCategoryDimensions` (line 325) creates a new 73-entry object literal on every call
  - `generateSIRepresentations` (line 681) wraps a library function but creates a new closure each render
  - `formatForClipboard` (line 567) is a complex function recreated every render
  - The `useEffect` dependency arrays are massive (line 323 has 13 dependencies)

- **`CONVERSION_DATA.find()` is called repeatedly** with O(n) scans:
  ```typescript
  // UnitConverterApp.tsx:146 -- on every render
  const getMassUnitInfo = (unitId: string) => {
    const cat = CONVERSION_DATA.find(c => c.id === 'mass');
    const unit = cat?.units.find(u => u.id === unitId);
    ...
  };
  
  // UnitConverterApp.tsx:178 -- on every render
  const categoryData = CONVERSION_DATA.find(c => c.id === activeCategory)!;
  
  // UnitConverterApp.tsx:257-259 -- on every render
  const fromUnitData = categoryData.units.find(u => u.id === fromUnit);
  const toUnitData = categoryData.units.find(u => u.id === toUnit);
  const fromPrefixData = PREFIXES.find(p => p.id === fromPrefix) || ...;
  ```
  With 73 categories and up to 100+ units per category, these linear scans are executed on every render. Pre-indexing into a `Map<string, CategoryDefinition>` and `Map<string, UnitDefinition>` would eliminate this.

- **Individual `useEffect` hooks on array indices** (lines 686-698):
  ```typescript
  useEffect(() => { ... }, [calcValues[0], calcValues[1], calcOp1]);
  useEffect(() => { ... }, [calcValues[1], calcValues[2], calcOp2]);
  ```
  React compares array element references, not values. Since `calcValues` is replaced with `[...prev]` on updates, this works but is fragile and non-idiomatic. These should use the full `calcValues` array or be consolidated.

- **Context value not memoized** in `ConverterContext.tsx:75-80`:
  ```typescript
  const value: ConverterContextValue = {
    state, dispatch, flash, inputRef,
  };
  ```
  A new object is created every render, causing all context consumers to re-render. Wrap in `useMemo`.

### 5. Accessibility (a11y)

**Severity: Critical**

This is the weakest area of the application. For a tool that handles precise numeric input and output, accessibility is essential.

**Missing fundamentals:**

- **No `<main>` landmark role.** The `<main>` element exists in `home.tsx:26` but the calculator content inside it lacks any ARIA landmarks or region labels.

- **No ARIA labels on interactive controls.** The swap button, copy buttons, and calculator operation buttons have no accessible names:
  ```tsx
  // ConverterPane.tsx:252-259 -- icon-only button with no label
  <Button variant="outline" size="icon" onClick={swapUnits} data-testid="button-swap"
    className="rounded-full w-10 h-10 ...">
    <ArrowRightLeft className="w-4 h-4" />
    {/* No aria-label, no sr-only text */}
  </Button>
  ```

- **Only 1 `aria-*` attribute in all feature components.** Across `UnitConverterApp.tsx`, `ConverterPane.tsx`, `CalculatorPane.tsx`, and `DirectPane.tsx` -- totaling ~3,700 lines of JSX -- there is exactly 1 `aria-*` attribute. This is in the CalculatorPane. The Radix UI select/tooltip primitives provide some built-in ARIA, but custom interactive elements have none.

- **No skip-navigation link.** The page has a header followed by complex interactive content.

- **No `aria-live` region for conversion results.** When a user changes the input value, the result updates reactively but screen readers have no way to know the result changed.

- **Positive `tabIndex` values** are used throughout (`tabIndex={1}` through `tabIndex={7}`):
  ```tsx
  // ConverterPane.tsx:150
  <Input ... tabIndex={1} ... />
  // ConverterPane.tsx:169
  <SelectTrigger tabIndex={2} ... />
  ```
  Positive `tabIndex` values override the natural DOM order and create unpredictable tab sequences. Use `tabIndex={0}` (or omit it) and rely on DOM order instead.

- **No focus management after operations.** After swapping units, clearing the calculator, or switching tabs, focus is not programmatically managed. The `refocusInput` function uses `setTimeout(() => { inputRef.current?.focus(); }, 100)` (line 252-254) -- the 100ms delay is a race condition against browser layout.

- **Color contrast concern.** The muted-foreground color (`215 20% 65%` which is ~`#8B95A8`) on the dark background (`220 15% 10%` which is ~`#16181F`) may not meet WCAG AA contrast ratio requirements for small text, though this would need to be verified with a contrast checker.

- **The `maximum-scale=1` meta tag** (index.html:5) prevents users from zooming:
  ```html
  <meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1" />
  ```
  This is a WCAG 1.4.4 violation. Remove `maximum-scale=1`.

- CSP headers are not configured (though for a single-file HTML deployment, this is less relevant).

const DIMENSION_TO_EXPONENT_KEY: Record<string, string> = {
  length: 'm', mass: 'kg', time: 's', current: 'A',
  temperature: 'K', amount: 'mol', intensity: 'cd',
  angle: 'rad', solid_angle: 'sr',

export function dimensionsToExponents(dims: DimensionalFormula): Record<string, number> {
  const exponents: Record<string, number> = { m: 0, kg: 0, s: 0, A: 0, K: 0, mol: 0, cd: 0, rad: 0, sr: 0 };
  for (const [dimKey, expKey] of Object.entries(DIMENSION_TO_EXPONENT_KEY)) {
    const val = dims[dimKey as keyof DimensionalFormula];
    if (val) exponents[expKey] = val;
  }
  return exponents;
}

## PART 4: SUMMARY

### Findings by Severity

| Severity | Count | Key Issues |
|----------|-------|------------|
| **Critical** | 1 | Accessibility: missing ARIA labels, no live regions, positive tabIndex, zoom prevention |
| **Major** | 3 | God component (1,676 lines), parallel state systems, no memoization causing excess re-renders |
| **Minor** | 8 | Dependency mismanagement, duplicate data structures, non-null assertions, unused dependencies, heading hierarchy, WebKit-only scrollbar, dead code |
| **Suggestion** | 3 | Scalability improvements, single-source-of-truth for dimensions, cross-browser CSS |

### Top 5 Recommendations (Priority Order)

1. **Fix accessibility fundamentals.** Remove `maximum-scale=1`. Add `aria-label` to all icon buttons and interactive elements. Add `aria-live="polite"` to the result display. Remove positive `tabIndex` values. Convert clickable `<div>` elements to `<button>`. This is the only critical finding.

2. **Migrate UnitConverterApp state to the existing reducer system.** The reducers, actions, and selectors are already built and tested. Using them would eliminate the prop-drilling problem (68 props on CalculatorPane), enable proper memoization, and reduce `UnitConverterApp.tsx` from 1,676 lines to ~200.

3. **Consolidate the dimension-mapping to a single source of truth.** Replace the three copies of `getCategoryDimensions` with a single import from `categoryDimensions.ts`. Extract the dimension-to-exponent mapping into a utility function.

4. **Clean up dependencies.** Move test packages to `devDependencies`. Remove 10+ unused packages. Delete the `queryClient.ts` boilerplate.

5. **Add integration and E2E tests.** The unit test suite is excellent, but the full component interaction path (paste -> parse -> convert -> display) is untested. The Playwright config is ready; write 5-10 E2E scenarios covering the primary user flows.

### What's Done Well

- **Domain modeling** is outstanding. The dimensional analysis system (formulas, SI derived units, cross-domain matching, alternative representations) is sophisticated and correct.
- **Test coverage** is impressive: 1,536 tests covering edge cases, localization, formatting, and regression.
- **Modular utilities** in `lib/calculator/` and `lib/units/` are a textbook example of single-responsibility, testable functions.
- **Localization** supports 12 languages and 8+ number formats including CJK myriad notation -- rare for a calculator app.
- **Single-file deployment** via `vite-plugin-singlefile` is clever and appropriate for the use case.
- **Type safety** is strong with clean compilation and proper TypeScript usage throughout.
